# Stage 0 вЂ” Bring-up: Design

**Date:** 2026-05-07
**Stage:** 0 (Bring-up) вЂ” see `CLAUDE.md` В§ Build Stages
**Status:** Approved, ready for implementation plan

## Goal

Get an ESPHome YAML compiling and running on the DFRobot FireBeetle 2 ESP32-C6 (DFR1075), connected to Wi-Fi and visible in Home Assistant via the native API. Verify the OTA path works end-to-end. No sensors beyond Wi-Fi signal and uptime; no deep sleep; no battery monitoring. Those land in Stages 1вЂ“4.

## Non-goals (deferred to later stages)

- BME280 / IВІC bus вЂ” Stage 1
- Reed switch / `pulse_counter` вЂ” Stage 2
- `deep_sleep:` component, wake-source handling, `weather_station_deep_sleep` helper *wiring* вЂ” Stage 3
- Battery ADC on `IO1`, low-voltage cutoff вЂ” Stage 4
- Lightning detector, wind/vane, soil/UV вЂ” Stages 8вЂ“10

The HA helper toggle entity itself is *created* in Stage 0 (so it exists when Stage 3 wires it up), but no ESPHome `binary_sensor: homeassistant` consumes it yet.

## Environment & workflow

- **ESPHome host:** Home Assistant add-on.
- **Initial flash:** [web.esphome.io](https://web.esphome.io/) in Chrome on Windows (HA add-on can't reach USB on the user's PC). Hold `GPIO9` (BOOT) while plugging in USB-C to enter download mode.
- **Subsequent flashes:** OTA from the HA dashboard.
- **Repo в†” dashboard sync:** repo is source-of-truth; manual copy/paste into the dashboard editor when changes are ready to install.
- **Hardware status:** board in hand; Stage 0 verification runs in this session.

## YAML content (`weather_station.yaml`)

Approach **B** вЂ” strict minimum plus dev-time bench affordances:

```yaml
substitutions:
  device_name: weather-station
  friendly_name: "Weather Station"

esphome:
  name: ${device_name}
  friendly_name: ${friendly_name}

esp32:
  board: esp32-c6-devkitc-1
  variant: esp32c6
  framework:
    type: esp-idf

logger:
  level: DEBUG          # tighten to INFO for deployed device later

api:
  encryption:
    key: !secret api_encryption_key

ota:
  - platform: esphome
    password: !secret ota_password

wifi:
  ssid: !secret wifi_ssid
  password: !secret wifi_password
  fast_connect: true
  ap:
    ssid: "Weather Station Hotspot"
    password: !secret ap_password

captive_portal:

web_server:
  port: 80              # dev only; remove in a later stage

status_led:
  pin:
    number: GPIO15
    inverted: false     # confirm polarity on first flash; flip if LED reads inverted

button:
  - platform: restart
    name: "Restart"

sensor:
  - platform: wifi_signal
    name: "Wi-Fi Signal"
    update_interval: 60s

  - platform: uptime
    name: "Uptime"
```

### Notes on specific choices

- **`board: esp32-c6-devkitc-1` + `framework: esp-idf`.** Per `CLAUDE.md`, IDF has more mature C6 support in ESPHome than `arduino`. The board ID supplies framework / flash-size / partition defaults only вЂ” all DFR1075-specific pins must be set explicitly when used (none used in Stage 0).
- **`fast_connect: true`.** Pre-emptive optimisation for Stage 3 deep-sleep wakes; harmless in Stage 0.
- **`ap:` + `captive_portal:` fallback.** Allows Wi-Fi recovery without USB if creds rot or the router changes.
- **`web_server:` and `status_led:` are dev-time affordances.** Both stay enabled through development; both will be reviewed for removal once the device is deployed.
- **`status_led` polarity.** The DFR1075 wiki is the source of truth, but polarity is verified by observation on first flash. If the LED is on when it should be off, set `inverted: true`.
- **Why no `deep_sleep:` yet.** Deep sleep + wake-source plumbing belongs to Stage 3. Including it in Stage 0 means writing code we can't verify against the actual sensors it will gate.

## Secrets

`secrets.yaml` (gitignored, real values):

```yaml
wifi_ssid: "your-ssid"
wifi_password: "your-wifi-password"
ap_password: "<random 8+ chars>"          # fallback hotspot
ota_password: "<long random string>"      # 32+ chars
api_encryption_key: "<base64-32-bytes>"   # `openssl rand -base64 32` or generated by the dashboard
```

`secrets.example.yaml` (checked in): same keys, all values `"REPLACE_ME"`. Documents required keys without leaking real values.

`.gitignore`:

```
secrets.yaml
.esphome/
*.bin
```

The `.esphome/` and `*.bin` lines cover the case where the ESPHome CLI is run locally for any reason (build artifacts, downloaded firmware binaries).

## Workflow doc (`docs/workflow.md`)

Five sections:

1. **First-time flash.** Copy `weather_station.yaml` + `secrets.yaml` into the HA ESPHome dashboard. *Install в†’ Manual download в†’ Modern format* в†’ download `.bin`. Open [web.esphome.io](https://web.esphome.io/) in Chrome on Windows. Plug in the C6 via USB-C while holding `GPIO9` (BOOT) to force download mode. Click *Connect*, select the COM port, *Upload* the `.bin`. Reset the board after flash completes.
2. **Subsequent flashes (OTA).** Once the device is on Wi-Fi and visible in HA's ESPHome integration, the dashboard *Install в†’ Wirelessly* path works. No USB.
3. **Editing the YAML.** Repo is source-of-truth. Edit `weather_station.yaml` here in the repo, copy/paste into the dashboard editor when ready to install. Commit changes to git first; treat the dashboard as a build/deploy target.
4. **Creating the HA helper toggle.** Settings в†’ Devices & Services в†’ Helpers в†’ *Create Helper в†’ Toggle*. Name: `weather_station_deep_sleep`. Default: ON. Unused in Stage 0; consumed in Stage 3.
5. **Stage 0 verification gate.** The checklist below.

## Verification gate

Stage 0 is done when ALL of these are true:

- [ ] `weather_station.yaml` compiles in the HA dashboard with no errors or warnings.
- [ ] Device flashes successfully via web.esphome.io and reboots.
- [ ] Device joins Wi-Fi and appears in HA's ESPHome integration (auto-discovered).
- [ ] `Wi-Fi Signal` and `Uptime` entities show live values in HA.
- [ ] OTA round-trip works: change `friendly_name`, install wirelessly from the dashboard, see the new name in HA.
- [ ] Onboard LED (`GPIO15`) behaves as a status indicator. If it's on when it should be off, flip `inverted` and re-flash.
- [ ] `weather_station_deep_sleep` HA helper toggle exists (unused in Stage 0).

## Risks & open questions

- **`status_led` polarity.** Resolved by observation on first flash; the spec accommodates either polarity via the `inverted` field.
- **`board: esp32-c6-devkitc-1` is a generic stand-in for the DFR1075.** No DFR1075-specific behaviour is exercised in Stage 0 (no battery ADC, no 3V3_C control, no IВІC). Generic devkit defaults are fine here. Stage 1 onward must set DFR1075 pins explicitly per the wiki.
- **Web flashing on Windows + C6.** ESP32-C6 USB-CDC enumeration on Windows is generally smooth, but if the COM port doesn't appear, the BOOT-button download-mode entry usually fixes it. Documented in `workflow.md` В§ 1.
